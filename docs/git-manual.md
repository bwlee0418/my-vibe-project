# 비개발자를 위한 Git 매뉴얼

## Git이란?

Git은 파일의 변경 이력을 관리하는 도구입니다. 문서 작업을 할 때 "보고서_최종.docx", "보고서_진짜최종.docx", "보고서_최종_수정.docx"처럼 파일을 여러 버전으로 저장해본 적이 있으시다면, Git은 이런 버전 관리를 자동으로 해주는 도구라고 생각하시면 됩니다.

Git의 핵심 개념을 쉽게 설명하면 다음과 같습니다. **커밋(Commit)**은 "변경사항을 저장한다"는 뜻이고, **푸시(Push)**는 "저장한 내용을 인터넷(GitHub)에 올린다"는 뜻입니다. **브랜치(Branch)**는 "원본을 건드리지 않고 별도 공간에서 작업한다"는 뜻이고, **풀(Pull)**은 "다른 사람이 올린 변경사항을 가져온다"는 뜻입니다.

## 클로드 코드에서 Git 사용하기

클로드 코드를 사용하면 Git 명령어를 직접 외울 필요 없이, 자연어로 요청할 수 있습니다.

### 변경사항 확인하기

현재 어떤 파일이 변경되었는지 확인하고 싶을 때 아래와 같이 요청합니다.

```
지금 어떤 파일이 변경되었는지 확인해줘
```

```
마지막 커밋 이후 변경된 내용을 보여줘
```

클로드 코드가 `git status`와 `git diff` 명령어를 실행하고, 변경 내용을 이해하기 쉽게 설명해줍니다.

### 커밋하기 (변경사항 저장)

작업 내용을 저장(커밋)하고 싶을 때 아래와 같이 요청합니다.

```
지금까지 작업한 내용을 커밋해줘. 커밋 메시지는 의미 있게 작성해줘.
```

```
PRD.md 파일 변경사항만 커밋해줘. 메시지는 "PRD 초안 작성"으로 해줘.
```

### 푸시하기 (GitHub에 올리기)

커밋한 내용을 GitHub에 올리고 싶을 때 아래와 같이 요청합니다.

```
변경사항을 GitHub에 푸시해줘
```

```
커밋하고 바로 푸시까지 해줘
```

### 브랜치 만들기 (별도 작업 공간)

새로운 기능을 만들 때 원본을 보호하면서 작업하려면 브랜치를 사용합니다.

```
"feature/meeting-recorder"라는 이름의 새 브랜치를 만들어줘
```

```
새 브랜치에서 작업하고 싶어. 브랜치 이름은 적절하게 만들어줘.
```

작업이 끝나면 원본에 합칠 수 있습니다.

```
이 브랜치를 main에 머지해줘
```

### 이전 버전 확인하기

이전에 작업한 내용을 확인하고 싶을 때 아래와 같이 요청합니다.

```
최근 커밋 히스토리 5개를 보여줘
```

```
어제 커밋한 내용을 보여줘
```

## 자주 발생하는 문제와 해결 프롬프트

### "커밋할 변경사항이 없다"고 나올 때

```
왜 커밋할 게 없다고 나오는지 확인해줘. 변경된 파일이 있으면 추가해서 커밋해줘.
```

### 실수로 잘못 커밋했을 때

```
방금 커밋을 취소하고 싶어. 변경사항은 그대로 유지한 채로.
```

### 푸시가 거부될 때

```
푸시가 안 되는데 왜 그런지 확인하고 해결해줘
```

### 충돌(Conflict)이 발생할 때

```
머지 충돌이 발생했는데 어떻게 해결하면 되는지 알려주고, 해결해줘
```

### GitHub 로그인이 안 될 때

```
gh auth status로 로그인 상태를 확인해줘. 로그인이 안 되어 있으면 로그인 방법을 알려줘.
```

## 일상 업무에서 Git 활용하기

Git은 코딩뿐 아니라 문서 관리에도 유용합니다. 마크다운(.md) 파일로 문서를 작성하면 Git으로 변경 이력을 자동 추적할 수 있고, GitHub에서 팀원과 함께 확인하고 피드백을 주고받을 수 있습니다.

매일 작업을 마칠 때 한 번씩 커밋하는 습관을 들이면, 언제든 이전 버전으로 돌아갈 수 있어 안심하고 작업할 수 있습니다.
